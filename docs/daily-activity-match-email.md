# Daily activity match email (8am local)

This feature runs the activity match **once per day at 8am in each user’s local timezone** and emails the result from **Ausna Community <community@ausna.co>** when the user has **≥ 1** matched activity.

## What was added

- **Cron endpoint**: `GET /api/cron/daily-activity-match` (Vercel Cron → calls hourly)
- **Daily match computation (service role)**: `lib/explore/dailyExploreMatchService.ts`
- **Email sending (Resend)**:
  - Client + helpers: `lib/email/resend.ts`
  - Template: `lib/email/templates/dailyActivityMatchEmail.ts`
  - Sender: `lib/email/dailyActivityMatch.ts`
- **Test-only UI button** on `/explore`:
  - Server action: `sendDailyMatchEmailForCurrentUser()` in `app/explore/actions.ts`
  - UI: `components/explore/ExploreView.tsx`

## Required environment variables

### Resend

- `RESEND_API_KEY`: Resend API key
- `RESEND_FROM_EMAIL`: must be exactly like:
  - `Ausna Community <community@ausna.co>`

### Supabase (already required for background indexing)

- `SUPABASE_SERVICE_ROLE_KEY`: used by cron + service compute to bypass RLS safely server-side

### Cron security

- `CRON_SECRET`: shared secret
  - Vercel Cron automatically calls your endpoint with header `Authorization: Bearer ${CRON_SECRET}`

### Unsubscribe (optional)

- `UNSUBSCRIBE_SECRET`: secret used to sign one-click unsubscribe tokens (min 16 chars). If not set, `CRON_SECRET` is used.

### URL used in emails

- `NEXT_PUBLIC_SITE_URL`: recommended (e.g. `https://ausna.co`)
  - Fallback: `VERCEL_URL`

## Resend setup: “Ausna Community”

1. In Resend, add and verify the domain **`ausna.co`** (DNS: SPF/DKIM).
2. Create a sender identity:
   - **From name**: `Ausna Community`
   - **From email**: `community@ausna.co`
3. Set `RESEND_FROM_EMAIL="Ausna Community <community@ausna.co>"` in your environment.

## User timezone storage

Each user’s timezone is read from their **human portfolio** row:

- `portfolios.type = 'human'`
- `metadata.properties.timezone = "<IANA timezone>"` (example: `"Asia/Tokyo"`)

If timezone is missing or not set (e.g. no location recorded for the user), the cron uses **Tokyo time zone** (`Asia/Tokyo`) as the default so the user still receives the daily match at 8am JST.

## Vercel Cron setup

This repo includes `vercel.json`:

- Runs hourly: `0 * * * *`
- Calls: `/api/cron/daily-activity-match`

On each invocation, the endpoint checks each user’s local time and only runs when that user is at **8am** and hasn’t already run today (based on `metadata.properties.daily_explore_match.ran_at`). Users who have unsubscribed (`metadata.properties.daily_explore_match.unsubscribed === true`) are skipped and do not receive the email.

## How to test the daily match cron

1. **Call the cron endpoint** with your `CRON_SECRET` (same as Vercel Cron uses):
   - Header: `Authorization: Bearer <CRON_SECRET>`
   - Or query: `?secret=<CRON_SECRET>`
   - Example (local):  
     `curl -H "Authorization: Bearer YOUR_CRON_SECRET" "http://localhost:3000/api/cron/daily-activity-match"`
   - Example (preview/production):  
     `curl -H "Authorization: Bearer YOUR_CRON_SECRET" "https://your-domain.vercel.app/api/cron/daily-activity-match"`

2. **When does it actually send?**  
   The cron runs every hour. It only runs the match and sends email for users whose **local time is 8:00** (in their stored timezone, or Tokyo if none). To test without waiting for 8am:
   - **Non-production only:** add `&force_run=1` to the URL. This skips the 8am check and runs match + email for every eligible (non-unsubscribed) user.  
     Example:  
     `curl -H "Authorization: Bearer YOUR_CRON_SECRET" "http://localhost:3000/api/cron/daily-activity-match?force_run=1"`
   - `force_run` is ignored in production.

3. **Logs**  
   The handler logs with the `[daily-activity-match]` prefix: START, batch load, per-user skip reasons (not 8am, unsubscribed, already ran today), run match, send email, and DONE summary. Check your server or Vercel logs for these.

## Test-only “send email” button

## Unsubscribe

- **Headers**: Each daily match email includes `List-Unsubscribe` and `List-Unsubscribe-Post` (RFC 8058) so Gmail and other clients can show a one-click “Unsubscribe” action.
- **Footer**: The email body includes an “Unsubscribe” link that goes to the same flow.
- **Endpoint**: `GET /api/unsubscribe/daily-match?token=...` — verifies the signed token, sets `metadata.properties.daily_explore_match.unsubscribed = true` on the user’s human portfolio, and shows a confirmation page. `POST` to the same URL is supported for one-click unsubscribe (returns 202).
- **Token**: Generated by `lib/email/unsubscribeToken.ts` (HMAC-SHA256 of user id). No database table is required for tokens.
